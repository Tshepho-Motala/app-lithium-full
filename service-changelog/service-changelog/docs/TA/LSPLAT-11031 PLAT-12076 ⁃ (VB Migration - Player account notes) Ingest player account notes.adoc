== Architecture
=== Technical
NOTE: This process can only begin once all players have been created in lithium, with respective lithium GUID's.

IMPORTANT: The ingestion process should support resuming in case of failure of the process or interruptions on infrastructure. The data batches read from the source data should be tracked and the most recent incomplete batch should be the starting point when the ingestion is resumed. There should be care taken to avoid duplicate records being ingested during this process.

IMPORTANT: Any provided sequences will stipulate a happy path process. All failure scenarios should be thought through, handled, retried where possible, or ultimately fail and rolled back and placed in a parking lot, as per details on previous technical specifications.

TIP: Refer to https://gitlab.com/playsafe/lithium/app-lithium-full/-/blob/3887b3f2b0c8c2dca43259d0f3d008e2ce496e42/service-accounting/service-accounting-provider-internal/docs/TA/LSPLAT-9918-PLAT-11049-vb-migration-accounting-ingest-historic-transactions.adoc#user-content-prerequisites for a brief description of DLQ and PL used for handling failures. This link will become ineffective once the referenced branch above has been merged. After which, the document can be found in the develop branch in the same directory structure.

In this specification, the requirement is to record the player notes to the changelog service in the Lithium platform with the correct taxonomy and original timestamps.

==== Categorisation
The player account notes need to be ingested with the relevant categorisation for the note. +
There needs to be a mapping of the DK categories and Livescore categories to allow for the correct segmentations during the insertion of the account notes. +
When production dry-run happens, there could potentially be additional categories and sub-categories, so this should be expected and changes to the mapping configuration will need to be made to accommodate this.
[source,sql]
----
-- DK data
SELECT CategoryName, SubCategoryName
FROM `lithium-virginbet-sandbox.dk_files.CustomerNotes`
GROUP BY CategoryName, SubCategoryName
ORDER BY CategoryName asc, SubCategoryName asc
LIMIT 1000
----

[source, sql]
----
SELECT c.name as category, sc.name as subcategory
FROM lithium_changelog.sub_category sc
JOIN lithium_changelog.category c
ORDER BY c.name asc, sc.name asc
----

==== The dataset
The data will need to be ingested with the relevant creation and modification date so that the note dates correlate to the original capture timestamp of the event. +
There might not be relevant categories for some items from the DK data import and a decision would need to be made on the correct attribution of a category for those. +
The 'isImportant' field in DK data will need to be mapped to the 'pinned' field in the lithium changelog table. +
The 'isDeleted' and 'DeletedDate' in DK data will need to be mapped to 'deleted' field and 'change_date' field. +
This will have the effect of the change date becoming the DeletedDate instead of the change date for the relevant changelog entry.
[source,sql]
----
SELECT CustomerID, CategoryName,SubCategoryName, isImportant, CreationDate, isDeleted, DeleteDate, Note
FROM `lithium-virginbet-sandbox.dk_files.CustomerNotes`
ORDER BY CreationDate ASC
LIMIT 1000
----

=== The process
==== Convert DK note category and sub category to lithium changelog category and subcategory

Given the existing Lithium categories and subcategories, the DK ones should be mapped to the equivalent ones in Lithium or new ones should be added in the unlikely event that there is no mapping that matches. +
See the classes: +
lithium.service.changelog.data.entities.Category +
lithium.service.changelog.data.entities.SubCategory


==== Ingest the player account notes
Use the same ingestion auditing capture that would have been used for the user ingestion. There would have been a need to also capture the changelog entry for the original user registration activity. If this was not needed or completed, the need would be there to create a changelog entry registration endpoint that would need to capture the relevant account note and allow for the allocation of the creation date. +
The main modification will be to expose the lithium.service.changelog.data.entities.ChangeLog.changeDate attribute in the ChangeLog entity. This adaptation will allow for the injection of the original date into the changelog entry. +

IMPORTANT: The capturing endpoint should be a new endpoint and there should be no change to the existing changelog processes or the functionality. Running the risk of unexpected behaviour on the current changelog implementation workflows would be catastrophic.

See the classes: +
lithium.client.changelog.ChangeLogClient +
lithium.client.changelog.ChangeLogService +
lithium.service.changelog.data.entities.ChangeLog +
